# Quel prompt utiliser pour transformer une liste de concurrents en insights produit

Beaucoup d'équipes ont déjà des concurrents en tête.

Le vrai blocage n'est pas de trouver un outil de plus.

Le vrai blocage est plus simple :
**quoi demander exactement pour obtenir une sortie utile ?**

Souvent, les gens partent avec une intention correcte, mais une demande trop floue :

- "analyse mes concurrents" ;
- "regarde ce que font ces apps" ;
- "dis-moi ce qu'on devrait copier".

Ce type de prompt produit rarement des insights produit propres.

Il manque généralement :

- le projet concerné ;
- le ou les concurrents à suivre ;
- le type de sortie attendue ;
- les sources de feedback à croiser ;
- et la distinction entre faits, signaux et idées roadmap.

La bonne nouvelle, c'est qu'il ne faut pas écrire un brief compliqué.

Il suffit de donner aux bons skills une demande plus structurée.

## Pourquoi un mauvais prompt produit des insights trop vagues

Quand la demande est trop large, la sortie l'est aussi.

Vous obtenez alors :

- une description générique du concurrent ;
- une liste de features peu priorisée ;
- des observations marketing superficielles ;
- et très peu de matière utile pour une vraie décision produit.

Le problème n'est pas seulement la qualité du texte.

Le problème est que vous ne forcez pas assez la demande à remonter :

- les retours utilisateurs réels ;
- les frictions récurrentes ;
- les points forts concrets ;
- les écarts entre promesse marketing et expérience vécue ;
- et les hypothèses qui peuvent nourrir votre roadmap.

## Les informations minimales à donner

Pour obtenir quelque chose d'utile, vous n'avez pas besoin d'un dossier complet.

Le minimum, c'est :

1. le nom de votre projet ;
2. 1 à 5 concurrents maximum ;
3. des noms d'apps ou des liens ;
4. l'objectif de la sortie ;
5. les sources de feedback à croiser si vous en avez en tête.

Par exemple, cela peut être :

- un site officiel ;
- une page AppSumo ;
- une page Play Store ;
- un profil Trustpilot ;
- ou simplement un nom d'application à retrouver.

Autrement dit, vous ne faites pas la veille à la main.

Vous donnez juste assez de matière aux skills pour qu'ils sachent quoi enrichir.

## Le prompt simple pour transformer des concurrents en insights produit

Si vous voulez une base propre, voici le prompt le plus simple :

```text
Pour [nom du projet], analyse ces concurrents :
- [Concurrent 1]
- [Concurrent 2]
- [Concurrent 3]

Retrouve leurs sources officielles, classe-les correctement, puis enrichis avec feedback utilisateur si disponible.
Sépare clairement :
- faits produit confirmés ;
- signaux clients récurrents ;
- idées roadmap possibles pour notre produit.

Indique aussi quels concurrents méritent une veille plus approfondie.
```

Ce prompt marche bien parce qu'il force déjà les skills à :

- identifier les bons concurrents ;
- retrouver les bonnes surfaces ;
- croiser promesse et feedback ;
- et produire une sortie exploitable, pas seulement descriptive.

## Variante : le prompt orienté feedback utilisateur

Si votre priorité est de comprendre ce que les utilisateurs aiment, critiquent ou demandent, utilisez une variante plus précise :

```text
Pour [nom du projet], prends ces concurrents :
- [Concurrent 1]
- [Concurrent 2]
- [Concurrent 3]

Va chercher leurs retours utilisateurs sur les meilleures sources disponibles
(AppSumo, Play Store, App Store, Trustpilot, G2, Capterra ou autres surfaces pertinentes).

Je veux :
- les frictions qui reviennent ;
- les fonctionnalités adorées ;
- les objections fréquentes ;
- le vocabulaire réel des utilisateurs ;
- et les signaux qui pourraient améliorer notre produit.
```

Cette version est idéale si vous voulez nourrir :

- l'UX ;
- l'onboarding ;
- la roadmap ;
- la FAQ ;
- ou la copy produit.

## Variante : le prompt orienté roadmap

Parfois, vous ne cherchez pas d'abord une synthèse concurrentielle.

Vous cherchez une sortie plus décisionnelle.

Dans ce cas, demandez ceci :

```text
Pour [nom du projet], analyse ces concurrents :
- [Concurrent 1]
- [Concurrent 2]
- [Concurrent 3]

Croise site officiel + feedback utilisateur réel si disponible.
Transforme ensuite l'analyse en décisions produit avec 3 sections :
- frictions à corriger chez nous ;
- expériences à améliorer ;
- idées à tester plus tard.

Je veux une sortie orientée priorisation, pas une simple description des concurrents.
```

L'intérêt de cette variante est simple :
elle pousse directement les skills vers une matière exploitable en backlog.

## Quels skills utiliser selon la question

Le bon prompt aide, mais le bon routage compte aussi.

Pour ce cluster, les choix les plus logiques sont :

- `205-sf-veille` pour capter la source, qualifier le concurrent et préparer la veille ;
- `008-sf-end-user` pour comprendre les frictions, attentes et retours vécus ;
- `204-sf-market-study` pour le positionnement, la demande et les objections ;
- `408-sf-audit-gtm` pour comparer promesse marketing et perception réelle.

## Ce que vous devez attendre en sortie

Un bon résultat ne doit pas ressembler à une fiche Wikipédia.

Il doit vous donner :

1. les concurrents confirmés et leurs bonnes sources ;
2. leur type : direct, indirect, adjacent, inspirationnel ;
3. les points forts qui reviennent ;
4. les frictions signalées par les utilisateurs ;
5. les surfaces de feedback qui méritent une deuxième passe ;
6. les hypothèses produit à surveiller ;
7. les idées roadmap à valider.

L'important n'est pas d'obtenir "beaucoup d'analyse".

L'important est d'obtenir quelque chose que vous pouvez réutiliser :

- dans votre veille ;
- dans vos arbitrages produit ;
- dans votre réflexion UX ;
- et dans vos futures demandes aux skills.

## Les erreurs à éviter

Même avec un bon angle, certaines demandes restent faibles.

Évitez de :

- donner 15 concurrents d'un coup ;
- demander seulement "une analyse complète" ;
- mélanger benchmark, UX, pricing, copywriting et roadmap dans une seule demande confuse ;
- oublier de demander les sources de feedback réelles ;
- demander "quoi copier" au lieu de demander "quels signaux méritent d'être testés".

Le but n'est pas de produire un rapport impressionnant.

Le but est de produire de meilleurs choix produit.

## Le réflexe à garder

Vous n'avez pas besoin d'arriver avec un dossier déjà préparé.

Le bon réflexe est :

1. donner 1 à 5 concurrents ;
2. préciser le projet concerné ;
3. demander explicitement une sortie orientée feedback, produit ou roadmap ;
4. laisser les skills enrichir les bonnes sources ;
5. lancer ensuite une deuxième passe si un concurrent mérite d'être creusé.

En pratique, le plus important est de partir d'une demande simple mais bien cadrée.

C'est ce qui permet de transformer une liste de concurrents en vrai matériau produit, au lieu de rester avec une veille trop vague pour être utile.
