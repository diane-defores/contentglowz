# Comment transformer une liste de concurrents en insights produit et idées roadmap actionnables à partir de vrais retours utilisateurs

La plupart des équipes ont déjà une première liste de concurrents.

Cette liste existe souvent quelque part :

- quelques bookmarks ;
- une note Notion ;
- deux captures d'écran ;
- un message Slack avec un lien ;
- et parfois une liste mentale jamais vraiment tenue à jour.

Le problème, ce n'est pas seulement le désordre.

Le vrai problème, c'est que cette liste reste rarement transformée en matière utile pour le produit :

- vous voyez des concurrents, mais pas les problèmes réels de leurs utilisateurs ;
- vous repérez des features, mais pas les frustrations qui les entourent ;
- vous collectionnez des liens, mais pas des signaux exploitables ;
- et votre roadmap se nourrit plus d’intuition que de feedback réel.

Chez nous, la valeur ne vient pas du fait de tenir une liste à la main.

Le bon réflexe est beaucoup plus simple :
vous donnez **un nom d’application** ou **un lien**,
et les skills ShipGlowz s’occupent de transformer ça en source d’insights produit exploitable.

En public, on peut appeler ça un **radar concurrentiel** ou une **base concurrentielle**.
En interne, les skills alimentent bien un registre produit structuré que vous pouvez ensuite enrichir et réutiliser.

## Créer un radar concurrentiel utile à partir d’une simple liste

Une simple liste de concurrents n’est pas encore un actif produit.

Elle devient utile quand les skills la transforment en base vivante capable de garder :

- le nom du concurrent ;
- son lien officiel ;
- son type ;
- son intérêt pour votre produit ;
- les features ou workflows à surveiller ;
- les signaux utiles pour la roadmap ;
- et, quand c’est pertinent, les retours clients associés.

Autrement dit, la liste cesse d’être un pense-bête et devient une mémoire produit partagée.

## Pourquoi ajouter ses concurrents dans une base commune change la qualité des insights

Beaucoup d’équipes veulent directement trouver des idées produit chez les concurrents.

Mais avant cela, il faut répondre à une question beaucoup plus simple :

**quels concurrents voulez-vous suivre proprement ?**

Si cette base n’existe pas, le reste devient fragile :

- la veille devient opportuniste ;
- les comparaisons changent chaque semaine ;
- les retours utilisateurs restent dispersés ;
- et les insights ne s’accumulent pas proprement.

La structuration de la liste est donc l’étape d’avant. Ce n’est pas la finalité, mais c’est ce qui permet ensuite d’extraire de vrais signaux actionnables.

## Ajouter des concurrents sans bricoler un tableau à la main

Le gros avantage de notre approche, c’est que vous n’avez pas besoin :

- d’ouvrir dix onglets ;
- de copier-coller des URLs dans plusieurs outils ;
- de créer un tableau à la main ;
- de décider seul où ranger chaque concurrent ;
- ni de faire vous-même la première couche d’analyse.

Vous pouvez démarrer avec :

- le nom d’une application ;
- un lien AppSumo ;
- une page Play Store ;
- un site officiel ;
- un concurrent cité dans une conversation ;
- ou même une vague intuition du type "cet outil nous ressemble".

Les skills vont ensuite :

- retrouver la source principale ;
- classifier le concurrent ;
- l’ajouter au bon registre ;
- noter pourquoi il est pertinent ;
- rattacher les surfaces où vivent les retours utilisateurs ;
- et préparer la suite de la veille.

## Vous partez d’un nom ou d’un lien, pas d’un dossier complet

Il ne faut pas penser :

- "je dois déjà avoir toute ma liste prête" ;
- "je dois d’abord faire la recherche" ;
- "je dois collecter tous les liens moi-même".

Il faut penser :

- "je donne le nom de l’app" ;
- "je donne le lien que j’ai sous la main" ;
- "je demande aux bons skills de l’ajouter et de l’enrichir".

C’est précisément là que ShipGlowz doit faire le travail.

## Quels skills ShipGlowz utiliser pour enrichir une liste de concurrents

Pour ce besoin, la route la plus naturelle est :

- `205-sf-veille` pour capter la source, classifier le concurrent et l’ajouter au bon registre ;
- `204-sf-market-study` si vous voulez déjà cadrer demande, positionnement et valeur concurrentielle ;
- `008-sf-end-user` si vous voulez ensuite comprendre l’expérience réelle vécue par les utilisateurs ;
- `408-sf-audit-gtm` si vous voulez confronter promesse marketing et perception réelle.

Mais pour partir d’une liste brute et la transformer en base exploitable, `205-sf-veille` est le point d’entrée le plus logique.

## Comment ajouter une liste de concurrents à enrichir avec de vrais retours utilisateurs

### Cas 1 : vous avez juste un nom d’application

```text
Ajoute [Nom de l’application] à mon registre concurrent pour [projet].
Trouve la bonne source officielle, classe-la, et note pourquoi elle est pertinente.
```

### Cas 2 : vous avez déjà un lien

```text
Ajoute ce concurrent à mon registre pour [projet] :
[URL]
Classe-le correctement et note les signaux utiles pour la roadmap.
```

### Cas 3 : vous voulez déjà préparer la suite

```text
Ajoute [Nom ou URL] à mon registre concurrent pour [projet],
puis enrichis avec site officiel + feedback client si disponible.
```

### Cas 4 : vous avez plusieurs concurrents d’un coup

```text
Ajoute ces concurrents à mon registre pour [projet] :
- [Nom ou URL 1]
- [Nom ou URL 2]
- [Nom ou URL 3]

Classe-les par priorité et indique ceux qu’il faudra creuser ensuite.
```

## Transformer une liste de concurrents en signaux produit exploitables

Quand vous lancez ce type de demande, vous attendez des skills qu’ils :

1. identifient le concurrent ;
2. retrouvent la bonne page source ;
3. décident s’il est direct, indirect ou inspirationnel ;
4. l’inscrivent dans le registre pertinent ;
5. retrouvent les surfaces de feedback utiles ;
6. expliquent pourquoi il mérite d’être suivi ;
7. proposent, si nécessaire, la suite logique.

La suite logique peut être :

- un benchmark produit ;
- une analyse du feedback client ;
- une extraction d’idées roadmap ;
- une analyse de copywriting ;
- une veille de positionnement.

Le point important est celui-ci : la liste n’a de valeur que si elle débouche sur des patterns, des irritants récurrents et des opportunités concrètes pour votre produit.

## Ce qu’il ne faut pas faire

Évitez de :

- garder les concurrents seulement dans votre tête ;
- traiter chaque nouveau concurrent comme une recherche isolée ;
- mélanger concurrents directs, outils adjacents et simples inspirations sans les qualifier ;
- attendre d’avoir une liste parfaite avant de commencer ;
- faire manuellement un travail que les skills peuvent absorber.

Le registre n’a pas besoin d’être parfait au départ.

Il a besoin d’exister, d’être structuré, et d’être enrichissable.

## Comment obtenir des idées roadmap actionnables à partir de concurrents bien suivis

Une roadmap devient plus solide quand elle s’appuie sur des références stables et sur du feedback réel.

Avec une liste bien enrichie, vous pouvez :

- suivre les mêmes acteurs dans le temps ;
- accumuler les signaux au lieu de les reperdre ;
- comparer les plaintes, attentes et usages qui reviennent ;
- décider plus vite quoi ignorer, quoi surveiller et quoi tester.

Vous passez alors d’une simple liste de concurrents à une veille cumulative nourrie par des preuves terrain.

Et c’est cette accumulation qui permet de faire remonter de vraies idées roadmap actionnables plutôt que des envies vagues de copier une feature.

## Passer du radar concurrentiel au feedback utilisateur réel

Une fois vos concurrents ajoutés proprement, l’étape suivante n’est pas de relire leurs landing pages à la main.

L’étape suivante est de demander aux skills d’aller chercher les retours clients réels :

- AppSumo ;
- Play Store ;
- App Store ;
- Trustpilot ;
- G2 ;
- Capterra ;
- founder Q&A ;
- autres surfaces où les utilisateurs disent ce qui les aide ou les bloque.

C’est exactement la suite logique pour transformer un simple concurrent en source d’insights produit actionnables.
