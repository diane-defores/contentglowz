# Tech-Spec: Psychology Engine — Creator Brain + Customer Brain + Bridge

**Created:** 2026-02-21

## Overview

### Problem Statement

Content generated by the robot agents (SEO, newsletter, articles, scheduling) lacks authentic personality and marketing intelligence. It reads like generic AI output because there's no system connecting the creator's identity and worldview to customer psychology and content strategy. Every piece of content sounds the same regardless of who the creator is, what they're going through, or who their audience is.

### Solution

Build a bidirectional psychology engine with three layers:

1. **Creator Brain** — A weekly ritual page where the creator inputs raw thoughts across 5 streams (current work, worldview, emotional state, tools, marketing ideas). A CrewAI agent synthesizes these into an evolving narrative profile organized by chapters. The creator validates every narrative update before it becomes truth.

2. **Customer Brain** — Multi-persona models representing customer segments with pain points, desires, objections, and emotional triggers. Refined weekly via manual input and Google Analytics engagement data.

3. **The Bridge** — Crosses the creator narrative with customer pain to propose 2-3 personality-driven content angles. The creator picks an angle, and it becomes the writing guideline fed to all content agents.

### Scope

**In Scope:**
- Creator Brain: standalone weekly ritual page, 5 input streams, CrewAI narrative synthesis agent, validation UI, chapter-based narrative storage in Turso
- Customer Brain: multi-persona data model in Turso, weekly input interface, Google Analytics integration for adaptive refinement
- Bridge: angle generation engine (creator narrative × customer pain → suggested angles), selection UI, guideline output format for content agents
- Agent integration: multi-format renderer connecting to existing SEO/newsletter/article/scheduling robots
- Data layer: Turso via Drizzle (new tables), FastAPI endpoints for Python agent access
- All data scoped per-project (existing Project model)

**Out of Scope:**
- Passive/unconscious pattern analysis of creator behavior
- Real-time (daily) input cadence — weekly only
- Mobile app UI
- Social media analytics beyond Google Analytics
- A/B testing of content angles
- Direct Google Analytics dashboard (just data ingestion)

### Technical Decisions

1. **Weekly cadence for both brains** — sustainable, prevents data noise
2. **Narrative storage model** — creator identity stored as evolving story with chapters, not scores or tags
3. **AI-synthesized, human-validated** — CrewAI agent proposes narrative updates, creator approves/rejects before commit
4. **Multi-format output** — one narrative source of truth, rendered into purpose-built extracts per content agent
5. **Turso via Drizzle** — consistent with all existing project data, agents access via FastAPI endpoints
6. **Standalone ritual page** — dedicated weekly check-in experience at `/dashboard/ritual`, not a tab
7. **Suggested angle engine** — bridge proposes 2-3 angles, creator picks, becomes content guideline
8. **Project-scoped** — all psychology data tied to existing Project model
9. **Lazy agent loading** — import CrewAI agents inside FastAPI endpoints, not at module level
10. **Background tasks** — narrative synthesis and angle generation run as FastAPI BackgroundTasks

### New Database Tables

**`creatorProfile`** — One per user per project, the narrative source of truth
- id, userId, projectId, currentNarrative (text/json), currentChapter (text), status, createdAt, updatedAt

**`narrativeChapter`** — Chapters in the creator's story
- id, profileId, title, theme, summary, emotionalTone, startDate, endDate (nullable = active), createdAt

**`creatorEntry`** — Weekly raw inputs
- id, profileId, weekOf (date), currentWork, worldview, emotionalState, toolStack, marketingIdeas, createdAt

**`narrativeUpdate`** — AI-proposed narrative changes pending validation
- id, profileId, entryId, proposedNarrative, proposedChapterChanges (json), status (pending/approved/rejected), createdAt, reviewedAt

**`customerPersona`** — Customer segment profiles
- id, userId, projectId, name, segment, painPoints (json), desires (json), objections (json), emotionalTriggers (json), naturalLanguage (json), behaviorData (json), confidenceScores (json), status, createdAt, updatedAt

**`contentAngle`** — Generated and selected angles
- id, userId, projectId, profileId, personaId, angles (json array of 2-3), selectedAngleIndex, contentType, guideline (text), status, createdAt

### Acceptance Criteria

#### Creator Brain

- [ ] AC 1: Given a logged-in user with a project, when they navigate to `/dashboard/ritual`, then they see the 5-stream weekly input form with placeholders and a date picker defaulting to current week
- [ ] AC 2: Given a user fills all 5 input streams and clicks Submit, when the entry is saved, then the system stores the entry in `creatorEntry` table and triggers narrative synthesis in the background
- [ ] AC 3: Given narrative synthesis completes, when the user views the ritual page, then they see a pending narrative update with the proposed changes compared to their current narrative
- [ ] AC 4: Given a pending narrative update, when the user clicks Approve, then the `creatorProfile.currentNarrative` is updated and the update status changes to 'approved'
- [ ] AC 5: Given a pending narrative update, when the user clicks Reject, then the update status changes to 'rejected' and the current narrative remains unchanged
- [ ] AC 6: Given the Creator Psychologist detects a chapter transition, when the narrative update is proposed, then it includes a new chapter suggestion with title, theme, and emotional tone
- [ ] AC 7: Given an approved chapter transition, when the profile is updated, then the previous chapter's `endDate` is set and a new chapter record is created

#### Customer Brain

- [ ] AC 8: Given a logged-in user with a project, when they access the persona editor, then they see all active personas for that project with name, segment, and confidence indicators
- [ ] AC 9: Given a user creates a new persona with name, segment, and at least one pain point, when they save, then the persona is stored in `customerPersona` with all JSON fields properly structured
- [ ] AC 10: Given a user edits a persona's pain points or desires, when they save, then the persona is updated in the database and the list refreshes
- [ ] AC 11: Given a user deletes a persona, when they confirm deletion, then the persona status is set to 'archived' and it disappears from the active list
- [ ] AC 12: Given Google Analytics is configured, when persona refinement is triggered, then the Audience Analyst agent enriches the persona with behavioral data and updates confidence scores

#### Bridge (Angle Engine)

- [ ] AC 13: Given a creator with an active narrative and at least one persona, when they select a persona and content type and click Generate, then the Angle Strategist produces 2-3 distinct angles
- [ ] AC 14: Given generated angles, when the user views them, then each angle shows title, positioning, tone, key message, and robot guideline
- [ ] AC 15: Given generated angles, when the user selects one, then the selection is saved to `contentAngle` with the `selectedAngleIndex` and the generated guideline
- [ ] AC 16: Given a selected angle, when any content robot (SEO, newsletter, article) requests a narrative extract, then the extract includes the selected angle's guideline adapted to that robot's format

#### Error Handling

- [ ] AC 17: Given the FastAPI backend is unavailable, when narrative synthesis or angle generation is triggered, then the UI shows a user-friendly error message and the entry is still saved locally
- [ ] AC 18: Given a user without a creator profile, when they first visit the ritual page, then a new profile is auto-created for their current project
- [ ] AC 19: Given an unauthenticated request to any psychology API route, when the request is processed, then it returns 401 Unauthorized

#### Data Integrity

- [ ] AC 20: Given all psychology data, when queried, then every record is scoped to a specific userId and projectId — no cross-user or cross-project data leakage
- [ ] AC 21: Given a narrative update is approved, when the profile is updated, then the previous narrative version is preserved in the `narrativeUpdate` history (append-only)
